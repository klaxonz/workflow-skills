# Code Build

需求 → 设计 → 实现 → 验证 → 收尾。

## 触发边界

触发（用户要求新增，可提取验收标准）：
- 新增功能 / 模块 / 组件 / API
- 补写测试 / 补写文档
- 新增工具函数

不触发（交给 code-fix）：报错、traceback、启动失败、测试失败、回归、"这里坏了"。模糊时判断主意图，无法判定问用户。

## 核心方法

所有构建都遵循：**理解需求 → 实现 → 验证。**

### 理解需求
用户口头描述时，用 1-2 句话复述确认理解。提取验收标准：做完后应该能验证什么。

### 实现
**先读再写。** 读目标文件和它的上下文——代码风格、命名习惯、已有的抽象和工具函数。新代码应该融入现有代码，而非自成一体。改前确认不在红线列表。每次改后自查是否与需求一致。

### 验证
按项目实际工具执行 lint、类型检查、涉及模块的测试。无对应工具则跳过。**新增代码没有覆盖的路径补测试。**

逐条对照验收标准，不自行评估"手动验证通过"——请用户在实际环境确认。

完成时一句话告诉用户改了什么、验证结果、验收标准达成情况。

## 完整流程

**满足以下任一条件时，叠加完整流程：**
- 多模块联动（>2 个模块）
- 验收标准超过 3 条
- 需要跨会话追踪
- 用户明确要求按流程走

完整流程在核心方法之上叠加：

**需求文档** — 按 `_shared` 约定创建，验收标准单独列出请用户确认。

**设计文档** — 按 `_shared` 约定创建（`type: feature`），写：验收标准映射、涉及文件、复用检查、风险点、实现顺序。用户确认后开始编码。

**收尾** — 验证完成后，按 `_shared` 约定更新需求文档和设计文档状态。输出改动汇总。

**新问题** — 开发中发现的预存在 bug 创建 `source: code-build` issue，报告中列出，不在当前流程修复。问用户下一步。

## 约束

- 先设计后编码，方案用户确认后再动手
- 不改红线文件、不顺手重构无关模块
- 不提交，除非用户要求
